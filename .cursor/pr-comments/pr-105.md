## 🔴 Critical

### 1. Verificar acesso do usuário ao repo antes de usar installation token
**Arquivo:** `backend/src/controllers/ProjectController.ts:709-717`

O endpoint `connectRepo` não valida mais se o usuário tem acesso ao repositório. Um cliente pode fazer POST com qualquer `owner/repo` onde o app está instalado, e o controller resolve a instalação com credenciais do app e importa — incluindo repos privados que o usuário não pode acessar no GitHub.

**Correção sugerida:** Antes de chamar `GithubService.getRepoInstallation`, verificar com o user token se o usuário tem acesso ao repo.

---

### 2. Não confiar no userId do state do callback
**Arquivo:** `backend/src/server.ts:119-130`

O `state` é construído inteiramente no frontend com `btoa(...)`, e o callback roda sem sessão autenticada. Um round-trip OAuth forjado pode sobrescrever o `github_user_token` de outro usuário apenas trocando o `userId` no payload do state.

**Correção sugerida:** Validar o userId contra uma sessão autenticada, ou usar o token trocado para buscar o GitHub user e mapear ao usuário correto, ou assinar o state com HMAC.

---

## 🟠 Major

### 3. Manter contrato ModelResult nos helpers de token
**Arquivo:** `backend/src/models/UserModel.ts:370-397`

Os três métodos (`saveGithubToken`, `getGithubToken`, `clearGithubToken`) lançam erros crus e não verificam se a row foi afetada. Isso viola o padrão do projeto onde models retornam `ModelResult<T>`.

**Correção sugerida:** Alterar assinaturas para retornar `ModelResult<T>`, wrappear em try/catch, verificar se a row foi atualizada.

---

### 4. Adicionar timeout nas chamadas axios ao GitHub
**Arquivo:** `backend/src/services/githubService.ts:470-482, 510-518`

`listUserRepos()` e `getRepoInstallation()` são chamados de request handlers mas não têm `timeout`. Se o GitHub travar, as requests podem manter sockets ocupados.

**Correção sugerida:** Adicionar `timeout: 15000` nas chamadas axios.

---

### 5. Não mapear todo erro de installation lookup para "app não instalado"
**Arquivo:** `backend/src/services/githubService.ts:507-523`

`getRepoInstallation()` retorna `null` para erros transitórios, credenciais ruins, rate limits e "não instalado" — tudo vira a mesma mensagem 422.

**Correção sugerida:** Só retornar `null` para status 404; re-throw para outros erros.

---

### 6. Aguardar bootstrap do auth antes de redirecionar
**Arquivo:** `frontend/components/ProjectForm.tsx:44-45, 152-155, 172-173`

`useAuth()` pode ter `user === null` durante inicialização. Nesse intervalo, `beginGithubOAuthAuthorize(user?.id)` envia userId vazio, e o backend não consegue persistir o token → loop 401 → OAuth.

**Correção sugerida:** Usar `isLoading` do useAuth e só disparar redirect quando `user?.id` existir.

---

### 7. Recomputar activeBranch quando o projeto muda
**Arquivo:** `frontend/pages/ProjectDetail.tsx:319-330`

A lógica só inicializa a branch quando `activeBranch` é falsy. Navegando entre projetos, a branch anterior sobrevive e `loadCards()` pode consultar o novo projeto com uma branch inexistente.

**Correção sugerida:** Sempre recomputar branchToActivate quando branches são carregadas; limpar activeBranch no path 404.

---

### 8. Não condicionar "Gerar Flow" ao prop opcional onCardUpdated (outside diff)
**Arquivo:** `frontend/components/CardFeatureModal.tsx:554-565`

O botão "Gerar Flow" está condicionado a `onCardUpdated`, que é opcional. No fluxo atual do ProjectDetail, o modal abre sem `onCardUpdated`, então o botão nunca aparece.

**Correção sugerida:** Remover `onCardUpdated` da condicional; adicionar fallback local no `onSuccess`.
